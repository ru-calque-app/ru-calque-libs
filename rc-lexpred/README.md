# rc-lexpred

Лексическое знание ученика: попытки → компактное состояние → прогноз по новому тексту.

Отвечает на вопрос «сможет ли этот ученик передать смысл вот этого русского текста
по-английски — с лексической стороны». Грамматику, произношение и пунктуацию не оценивает.

## Что считается концептом

Единица учёта — **способность выразить смысл**, а не знание слова:

```
REJECT_OFFER  →  reject the offer / decline the offer / turn down the offer
```

Ученику достаточно уметь хоть одну реализацию. Обратная сторона: знание `make` и `decision`
по отдельности **не** означает знания `make a decision` — коллокация это самостоятельный
концепт со своей статистикой, а не производная от компонентов (тест
`knowing_the_component_words_does_not_make_a_collocation_known`).

Виды концептов: `lemma`, `phrase`, `collocation`, `phrasal_verb`, `idiom`, `family`.
Последний — узел для fallback, а не цель.

## Что крейт делает сам, а что обязан получить снаружи

Сам — только арифметику и бухгалтерию: веса, накопление, забывание, иерархический fallback,
компактизацию, сведе́ние в оценку. Всё детерминированно и проверяемо.

Снаружи приходят три вещи, которые детерминированно не выводятся:

| Что | Интерфейс | Почему не здесь |
|---|---|---|
| Требования текста | `LexicalExtractor` | переход «русский смысл → английский концепт» требует двуязычной онтологии; в ru-calque идёт через канонические переводы |
| Суждение о смысле | поле `meaning_conveyed` в `Attempt` | «донёс ли смысл перифразом» — суждение, а не факт |
| Каталог концептов | `ConceptCatalog` | описания, реализации, группы, prior'ы живут в БД/файле |
| Хранение состояния | `to_json` / `from_json` | крейт в БД не ходит |

Факт «форма прозвучала в ответе» крейт устанавливает **сам** словарём `rc-lex` — это факт,
а не мнение, и спрашивать о нём LLM незачем.

### Что должен дать extractor

Список `LexicalRequirement`: `concept_id`, `importance_weight` (важность для смысла ЭТОГО
текста, не сложность), `concept_kind`, `acceptable_realizations`.

⚠️ Требование **без единой реализации пропускается**: сопоставлять не с чем. Раньше такое
требование молча становилось «обходом» и приносило ученику 0.8 из воздуха.

### Что должен дать evaluator

Либо готовые `LexicalObservation` (`outcome` 0..1, `confidence` 0..1, время), либо —
предпочтительно — `Attempt { transcript, corrections }`, из которого крейт соберёт
наблюдения сам через `observe`. `Correction` — это `{item, correct, impact}`, ровно то, что
уже есть в `Report` из `ru-calque-eval`.

Сорванное распознавание речи должно ронять **`confidence`**, а не `outcome`: иначе ученик
получает провал за качество микрофона.

## Жизненный цикл состояния

```
загрузить состояние → применить наблюдения → сохранить состояние
загрузить состояние → получить требования → предсказать → показать
```

Состояние содержит только id концептов и свёрнутые свидетельства — ни сырых попыток, ни
исходных текстов, ни описаний концептов. Правка каталога не требует трогать состояния
учеников.

Размер ограничен `Config::max_concepts`; `compact_state` выбрасывает наименее полезные
записи. Выброс не теряет статистики: групповые агрегаты ведутся параллельно поконцептным.

Обновление и прогноз стоят по числу концептов в текущей попытке или тексте, а не по объёму
истории.

## Ограничения baseline sentence scorer

`expected_lexical_coverage` — честная величина: взвешенное среднее, интерпретируется
буквально.

`probability_lexically_acceptable` — **ранжирует, но не откалибрована**. `BaselineScorer` это
логистическая свёртка с коэффициентами, выставленными на глаз. Текст с 0.3 действительно
рискованнее текста с 0.8, но из «0.67» **не следует**, что в 67 случаях из 100 перевод
окажется приемлемым.

Произведением вероятностей эту величину называть нельзя: концепты зависимы, а ученик может
обойтись перифразом.

### Как подключить обученную калибровку

Скорер — трейт. Когда накопятся размеченные исходы (прогноз → реальная попытка → сверка):

```rust
struct CalibratedScorer { /* веса из обучения */ }

impl SentenceScorer for CalibratedScorer {
    fn score(&self, f: &SummaryFeatures) -> f64 { /* ... */ }
    fn name(&self) -> &'static str { "calibrated-v1" }
}
```

`SummaryFeatures` — ровно тот вектор признаков, который пойдёт в обучение. Остальной API не
меняется.

## Как добавить end-to-end сценарий

Положить JSON в `tests/fixtures/` — новый mock-класс не нужен, тест параметризован
каталогом. Структура: `concepts`, `attempts[{source, answer, observed_at, requirements,
corrections}]`, `target{source, requirements}`, `expect{...}`.

Затем принять эталон отчёта:

```bash
UPDATE_GOLDEN=1 cargo test -p rc-lexpred --test e2e
```

Проверяются и свойства (знакомый концепт выше проваленного, успех улучшает прогноз), и
точные числа через golden-файл `tests/golden/<name>.txt`.

## Что показал прогон на реальных данных

25 реальных попыток с прода (3 ученика, настоящая разметка колод и настоящие разборы)
прогнаны через весь путь. Механически всё работает. Про предсказательную силу ответ
неоднородный, и разделять его надо **по колодам**, а не по ученикам.

### Лексика циркулирует внутри колоды — но не во всякой

Доля единиц текста, встречающихся хотя бы в одном другом тексте той же колоды:

| Колода | Переиспользование |
|---|---|
| Backend Developer Interview | 53% |
| Собеседование на Elixir | 44% |
| Подготовка к интервью Elixir | 32% |
| Переезд в англоязычную страну | 9% |
| Смолток | 8% |
| Знакомства и small talk | **0%** |

Тематические и технические колоды дают модели материал: термины (`Elixir`, `Phoenix`,
`pattern`, `data`) ходят из текста в текст. Колоды на small talk — не дают вообще: там
каждый диалог про своё, и общей лексики между текстами нет по построению.

Насколько глубоко надо пройти колоду, чтобы материал появился (доля лексики нового текста,
уже встречавшейся ученику, при случайном порядке прохождения):

| Колода | k=3 | k=5 | k=8 | k=12 |
|---|---|---|---|---|
| Backend Developer Interview | 14% | 24% | 33% | 39% |
| Собеседование Elixir-разработчика | 22% | 27% | 34% | 42% |
| Смолток | 2% | 2% | 2% | 7% |

### Механизм работает там, где у него есть материал

На 20 отложенных случаях средняя ошибка прогноза:

| | n | ошибка |
|---|---|---|
| у концептов цели ЕСТЬ личная история | 7 | **0.15** |
| личной истории НЕТ | 13 | 0.22 |

То есть индивидуальный уровень действительно улучшает прогноз — примерно на треть.
Выборка крошечная, так что это указание, а не доказательство.

### Но на текущих данных общий прогноз всё равно вырожден

| | |
|---|---|
| Разброс прогноза | sd = 0.06 |
| Разброс факта | sd = 0.27 |
| Корреляция прогноз/факт (n=20) | ≈ 0 |

Причина не в модели, а в том, где находятся ученики: технический ученик прошёл 8 текстов
своей колоды из 20, поэтому реально пересеклось 13% лексики; двое других — на small-talk
колодах, где пересечение нулевое в принципе. При такой плотности индивидуальному уровню
неоткуда взяться, и всё сваливается в групповые агрегаты, почти одинаковые между текстами.

Это видно в фикстуре `prod_interview_prep`: все 10 концептов цели идут с `observations: 0`,
а `evidence_quality` честно равен `0.00`.

**Практический вывод:** модель осмысленна на тематических колодах и примерно с десятого
пройденного текста. На small-talk колодах она не заработает никогда — там нечему
переноситься, и показывать по ним лексическую готовность не следует.

### Гипотеза про группы не подтвердилась

`derived_groups` (семейство + частотная полоса + вид + уровень) добавлялись, чтобы знание
переносилось между текстами без общих лексем. На этих данных разброс прогноза не вырос, а
слегка сузился (0.063 → 0.057), корреляция осталась нулевой. Группы оставлены — они дёшевы
и логичны, — но пользы на имеющихся данных не показали, и выдавать их за решение нельзя.

### Оговорка про саму метрику

Факт здесь — «прозвучала ли точная единица», а это не то же самое, что «знает ли ученик
слово»: знающий `utilities` вполне скажет `bills`. Такой метрикой мы требуем от модели
предсказать ещё и выбор перифраза. Часть нулевой корреляции — на совести метрики. Честнее
мерить «донёс ли смысл» (`outcome ≥ 0.75`, успех или удачный обход).

n=20, факт квантован (0 / 0.5 / 0.67 / 1.0), поэтому «корреляция ≈ 0» означает «сигнала не
видно», а не «доказано отсутствие».

## Известные ограничения

- **Многословные реализации сопоставляются грубо**: у сочетания, которого нет в `rc-lex`,
  совпадение по леммам не смотрит на порядок и расстояние слов. Пиннится тестом
  `scattered_words_still_count_as_a_collocation`.
- **Цепочка постериоров — не строгий Байес**: одно наблюдение попадает и в концепт, и в его
  группы, и в общий счётчик. Осознанный размен интерпретируемости на формальную корректность.
- **`serde_json` не round-trip'ит f64 побитово** (расхождение ~1 ulp). Ни одно решение
  модели на этом не стоит, но требовать побитового равенства состояния после JSON нельзя.
- **Словообразовательные семейства неполны**: Портер знает только суффиксы, поэтому
  `develop`/`developer` сливаются, а `pay`/`payment`, `decide`/`decision`,
  `special`/`specialist` — нет. Пиннится тестом `porter_misses_many_derivations`.
- **Каталог концептов пуст**: `rc-lex` — словарь форм, коллокаций там нет и открытого
  источника для них не существует. Наполнение каталога — отдельная задача (курируемая, не
  генерация на лету: `concept_id` обязаны быть стабильными).
